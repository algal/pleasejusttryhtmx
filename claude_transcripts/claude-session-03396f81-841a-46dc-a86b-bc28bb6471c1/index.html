Pattern Analysis of Reference Sites

Common Style Patterns

  1. Aggressive, Profane Tone: All sites use profanity liberally as a rhetorical device—not gratuitous vulgarity but emphatic provocation that cuts through developer ego and creates memorable impact.

  4. Short, Punchy Sections: Headers, bullet points, brief paragraphs. Easy scanning. No walls of text.

  5. Hyperbolic Comparisons: Space shuttles made of popsicle sticks, Nazi WWII jokes, flamethrowers to light candles — absurd metaphors that make the point memorable.

  6. Self-Aware Irony: Acknowledging exceptions (some sites DO need complexity) while maintaining the core argument.

  7. Structural Progression:

  8. Open with aggressive hook
  9. State the problem with modern web dev
  10. Present the solution (their advocacy)
  11. Give concrete examples/proof
  12. Acknowledge edge cases
  13. Close with call to action

pleasejusttryhtmx.com enters this debate as a middle path: - "You're both wrong. You don't need raw HTML limitations AND you don't need React's complexity. HTMX gives you interactivity without the bullshit."

Key HTMX Arguments to Feature

The Killer Stats (from React→HTMX port)

  • 67% less code (21,500 → 7,200 lines)
  • 96% fewer JavaScript dependencies (255 → 9)
  • 88% faster builds (40s → 5s)
  • 50-60% faster initial load

Philosophy Points

  1. HTML is a hypermedia — we forgot this. HTMX "completes HTML."
  2. Why are only <a> and <form> allowed to make requests? Any element should be able to.
  3. Server returns HTML, not JSON — no API contract maintenance, no client-side rendering
  4. Locality of behavior — the logic is right there in the markup
  5. Full-stack simplicity — backend devs can build interactive apps without learning React
  6. 16kb — that's the whole library

Position in the Debate

  • To justfuckingusehtml crowd: "You're right that HTML is powerful, but sometimes you DO need dynamic updates without page reloads. HTMX gives you that with zero JavaScript writing."
  • To justfuckingusereact crowd: "You're solving imaginary problems. 90% of apps don't need component trees and virtual DOM diffing. HTMX handles real interactivity with HTML attributes."

Analysis

What the official demo.js does: - Uses Sinon.js (external library) to create a fake server - Provides a clean routing API: onGet('/path', handler), onPost('/path', handler) - Parses query params and form data automatically - Has an activity tracker UI that shows request/response logs visually on the page - 80ms response delay

What we have: - Pure JS fetch() interception, no dependencies - Simple key-based matching ('POST /demo/clicked') - Console logging for debugging - 150-200ms delays - Completely self-contained

Verdict: Keep what we have

Reasons:

  1. No external dependencies - Our approach fits the "no package.json" ethos of the page. Adding Sinon.js would undermine the message.

  2. Simpler for our use case - The official demo.js is built for htmx.org which has dozens of demos. We have 3. Our key-based approach is plenty.

  3. The activity tracker is tempting but wrong for us - It would be educational to show requests/responses visually, but it would clutter the page and distract from the main argument. Curious developers can open DevTools and see the console logs.

  4. Ours actually works with htmx 4 - The official demo.js intercepts XMLHttpRequest via Sinon. We intercept fetch(), which is what htmx 4 uses.

One small thing we could steal: The 80ms delay feels snappier. We could drop ours from 150-200ms to ~100ms. But that's minor.

Bottom line: We independently arrived at a similar solution, just lighter weight. Ship it.
